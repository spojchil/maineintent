# 动作—观察闭环与第一人称感知实验发现

本文汇总 D40 动作—观察闭环以及随后真人旁观视口实验中值得保留的发现。它是实验记录，不是新架构契约，也不把任何 `Dxx` 条目自动改成“已决定”。

实验代码从 Draft PR [#73](https://github.com/spojchil/maineintent/pull/73) 继续演化；本文核对的提交基线为 `experiment/d40-run@69bdb33`。该分支尚未形成可合并基线，后续实体碰撞箱、紧凑实体投影等调整仍处于未提交现场验证阶段，因此相关内容只记录现象和候选方向，不声称实现已经落地。

本文没有保存密钥、服务器地址、本机路径、真人用户名或原始私人聊天。公开材料只保留复现实验所需的场景、聚合指标和脱敏例句。

## 1. 实验回答了什么

D40 的最小场景是：

```text
玩家说“看看那只羊，然后走过去”
→ 模型获得当前第一人称视口
→ 调用相对转头或短时移动
→ Runtime 执行真实身体输入
→ 返回动作后的新视口
→ 模型继续调整或自然回应
→ 全程不由模型管理世界目标 ref
```

它同时检验四个问题：

1. 原生 Tool Calls 能否形成真正的“动作—观察—再决策”循环；
2. 相对视口是否足够支持转头和短距离接近目标；
3. 多轮模型往返的延迟和成本是否仍像一个可用的同伴；
4. 动作或感知失败后，模型能否根据真实结果收敛，而不是补写故事。

### 1.1 场景与证据

确定性场景把同伴放在平坦地面上，静止羊位于正前方约 12 格处，同伴初始朝向偏离约 64°。统一指令为“看看那只羊，然后走过去”。

结论来自三路相互独立的证据：

| 证据层 | 能回答的问题 | 边界 |
|---|---|---|
| 服务端场景与查询 | 实体是否移动、方块是否存在、最终距离 | 只作测试真值，不能进入生产认知 |
| Runtime 事件与工具轨迹 | 模型实际调用了什么、身体姿态怎样变化 | 当时没有记录每次工具返回的完整视口 |
| 真人或观察者脚本 | 动作是否真的发生、聊天与动作是否一致、旁观看起来是否自然 | 不替代服务端真值，也不适合高频回归 |

这延续了信息接口提案中的 O0/O1/O2 隔离：测试可以用上帝视角裁判生产输出，但不能把裁判信息喂回模型。

## 2. D40 的实测结果

### 2.1 闭环确实成立，但尚未证明架构已经选定

修复供应商请求格式后，真实模型能够连续执行：

```text
look_relative → 新视口 → move_input → 新视口 → 再次纠正
```

模型能把约 64° 的初始水平偏差修到几度，并在移动过冲前持续缩短与目标的距离。最好一次到达约 0.74 格，说明不向模型暴露实体 ID 或世界目标 ref，也能完成这个窄场景中的目标发现和接近。

这为 `D01` 提供了正向证据，但只证明短动作 Tool Loop **可行**。它没有证明所有 Grounding、内部目标锁定或长动作控制都应删除，也没有覆盖方块交互、GUI、战斗、导航和长期共同活动。

### 2.2 聚合数据

修复前两次运行都没有产生工具调用。修复后五次运行产生 5–14 次身体工具调用：

| 指标 | 观察值 |
|---|---|
| 单次总耗时 | 30.5–80.7 秒 |
| 单次输入 token | 123,273–280,109 |
| 最接近目标 | 0.74 格 |
| 有工具调用的最终环境报告 | 1 次经真值支持，4 次包含不实环境解释 |

这些数字不是性能基准：prompt、上下文重放和视口仍是实验实现。但它们足以否定“多轮往返天然只是可信的慢”这一未经验证的乐观判断。认知迟疑可能自然；几十秒的动作停顿、六位数上下文重放和迟到的危险反应并不自动自然。

把 `reasoning_effort` 调低也没有自动省钱。一次对照中，工具调用数增加约 56%，token 增加约 68%，耗时增加约 45%；模型只是把动作拆得更碎。因此推理深度不能代替闭环预算、上下文增量和本地运动控制。

## 3. 接口与供应商发现

### 3.1 工具阶段与最终结构化输出必须分开

实验最初同时发送 `tools` 和 `response_format: json_object`。在所用 DeepSeek-compatible 接口上，模型稳定返回普通 JSON 内容而不是 `tool_calls`，并会用文本声称自己已经观察或移动。

对照移除 `response_format` 后，相同模型和 prompt 立即返回 `finish_reason: tool_calls`。可工作的循环因此分成两阶段：

1. 工具阶段允许原生 Tool Calls，不强制最终 JSON；
2. 模型停止调用工具后，再关闭工具并取得最终结构化社交输出。

这补充了 `F11` 与 `D36`：供应商兼容不能只验证 schema 能否提交，还要用真实端点验证 `finish_reason`、Tool Calls、思考内容回传和最终输出阶段的组合行为。

### 3.2 模型不管理世界 ref，不等于 Runtime 不需要身份

D40 没有证明内部身份可以删除。Runtime 仍需要把供应商 `tool_call_id`、决策轮次、连接/世界/维度 epoch、取消信号和日志证据关联起来；长动作还可能需要 action ID、资源占用和短时目标锁。

实验支持的窄结论是：**模型不需要通过 opaque world-target ref 指认这只羊。**内部控制与审计标识属于另一层，不能由这个结论顺带删除。

## 4. 身体与调度发现

### 4.1 真实按键语义必要，但不能是唯一移动能力

`move_input` 只是按住前、后、左、右之一一小段时间。它证明了最低层身体输入可以工作，也暴露出三个问题：

- 视线与移动解耦，走近目标时很容易把目标甩出画面；
- 按时间移动难以在目标前稳定停下，最后一两格最容易过冲；
- 每个小步都经过模型网络往返，延迟和费用迅速放大。

后续不应删除低层输入，也不应要求它承担全部导航。更合理的实验方向是同时保留：

- 可观察、可取消的短时真实输入，用于细调、试探和失败恢复；
- 一个更高层但仍受合法观察、持续反馈和中断约束的移动能力，用于本地连续控制。

高级能力不应暗含未观察地图、瞬时完美路径或不可打断的任务宏。Pathfinder 是否可用应由信息来源、轨迹和失败行为判断，而不是仅由库名判断。

### 4.2 工具返回时，身体可能还没有稳定

一次运行中，`move_input` 返回的高度为 83，下一次工具开始前已经继续下落到 81。模型收到的是运动中途快照，却不知道自己仍在下落；这次落差又改变了目标仰角。

动作结果必须能区分：

- 输入已经释放；
- 身体仍在滑动、下落或被碰撞推动；
- 身体已经稳定；
- 世界产生了什么可观察效果。

这不是要求现在冻结 `ActionOutcome` 双轴 schema，而是说明“按键释放”不能被等同于“动作后的世界状态已经确定”。

### 4.3 只讨论聊天 FIFO 不够

D40 当前把主要玩家聊天按顺序交给模型，新的普通消息不会自动抢占正在运行的短动作。这避免了运行时把“停下”硬编码成口令，但长轮次期间还会到达声音、受伤、实体变化、其他玩家消息和连接事件。

未来需要的是统一的时序与注意力策略，而不是简单选择“所有东西都 FIFO”或“只有聊天 FIFO”：

- 玩家消息需要保序并防止静默丢失；
- 高频声音可能需要合并、过期和显著性排序；
- 受伤、死亡、断线和换维度会使旧执行上下文失效；
- 普通新信息可以等到检查点，由模型决定继续、暂停或回应；
- 本地危险反应不能等待完整网络思考循环。

这同时影响 `D12`、`D17`、`D28`–`D30` 和 `D35`，不应只作为聊天模块的队列细节处理。

## 5. 第一人称感知发现

### 5.1 圆锥不是 Minecraft 的画面

Minecraft 的 FOV 值是垂直视场，水平视场由画面宽高比推导；屏幕对应矩形视锥，不是单一夹角的圆锥。默认垂直 FOV 70°、16:9 下，水平视场约为 102.45°。

实验代码原先使用 45° 圆锥，`main` 一度使用更窄的 35° 圆锥。改为分别判断水平和垂直平面后，森林与墙面场景都找到了更多真实可见方块。视锥必须来自相机投影参数，而不能靠一个“看起来合理”的角度常量。

### 5.2 实体是否可见，不能只看中心点

真人附着到同伴第一人称视角时，屏幕中能看到羊和它的碰撞箱，但中心点投影仍可能落在垂直视锥之外，导致结构化视口返回空实体。近距离、大型或部分出屏实体都会遇到同一问题。

正确的几何对象至少是实体可见包围盒，而不是脚点或中心点：先做 AABB 与矩形视锥相交，再用包围盒上的多个样本检查方块遮挡。现场修改后，同一近距离场景能够返回羊；这项修改仍需在可复现分支上整理后才能算实现证据。

### 5.3 方块也不能只看中心点

平视纯平地面时，近处地面中心点落在垂直视锥外，远处地面又被前方同高度方块判为遮挡，最终可能返回 0 个方块；低头后才出现少量地面。

玩家看到的是方块暴露表面，而不是中心点。方块视锥、遮挡和射线判断应基于实际可见面或有界多点采样，并考虑非完整碰撞/渲染形状。实体 AABB 修复不能替代这项工作。

### 5.4 透明、半透明与形状不是一个布尔值

当前实验把非空气方块视作可见，再以 `transparentHint` 粗略决定它是否遮挡。这能表达“玻璃可见但不完全挡住后面”，却不能准确覆盖树叶、液体、栅栏、门、台阶和不同渲染层。

未来应分别考虑“这个方块自身是否可见”“它遮挡哪些采样射线”“它的可见/碰撞形状是什么”，并用版本化材料或形状事实驱动。不能把所有半透明方块统一当空气，也不能把所有非空气方块统一当整立方体。

### 5.5 最近 N 个方块不是完整视觉模型

把上限从 20 提高到 256 能减少过早截断，却没有解决选择问题：

- 纯石头平台会用大量同质近处方块占满结果；
- 远处树木、门口或建筑轮廓等地标可能完全挤不进来；
- 更多方块增加 token 和注意力负担，却不一定增加可用信息；
- 一次“数清整个平台”的要求本就接近视觉测量任务，不能只靠最近排序保证。

值得实验的是同一个视口接口提供近、中、远范围或分层配额，由模型根据当前问题选择，而不是增加“获取脚下方块”“找树”等对象专用工具。静态显著性、时间变化和视线中心排序仍是候选机制，不能未经测试直接固化。

### 5.6 相对坐标保留即时可操作性，却破坏跨帧物体恒常性

当前四元组 `["stone", right, up, forward]` 很适合回答“它现在在左边还是前面”。但每次转头都会旋转坐标系，每次移动都会平移原点。绕一棵树走几步后，模型必须仅凭离散文本快照自行完成空间配准，才能判断新画面里的树是否还是原来的树。

真人依赖连续画面、光流和身体运动感维持物体恒常性，当前模型没有这些输入。因此“只给相对坐标”未必更接近真人，反而可能删除真人原本拥有的连续证据。

绝对坐标应重新进入 `D16`、`D18`–`D21` 的实验选项。需要比较：

1. 仅身体相对坐标；
2. 相对坐标加稳定局部坐标；
3. 相对坐标加 Minecraft 世界坐标和观察者姿态。

无作弊边界应优先约束**信息来源与可见范围**：只给当前合法观察到的对象，不因坐标稳定就查询墙后或未加载世界。世界坐标是观察的表示方式，不自动等于世界查询或 opaque ref。它是否过于精确，仍需通过旁观体验和服务器信息规则验证。

### 5.7 紧凑四元组会丢失身份与可见状态

`["sheep", right, up, forward]` 无法区分普通羊、名为 sheep 的玩家和带可见名称的实体；`["oak_door", ...]` 也没有表达门是否开启，熔炉是否燃烧，作物处于哪个可见生长阶段。

下一版不应为每种对象增加专用字段，也不应暴露全部 NBT。应研究一个通用、紧凑的“可见描述符”，区分对象类别、显示名和客户端确实渲染或通常可见的状态。方块状态、实体元数据与名字只有在玩家侧可观察时才进入投影。

### 5.8 坐标约定必须接受独立几何测试

PR [#75](https://github.com/spojchil/maineintent/pull/75) 发现 `relativeBearing` 的左右分支完全颠倒，实体和声音方向都会镜像。此类错误往往不会崩溃，只会让同伴显得莫名糊涂。

所有相机轴、世界轴、yaw/pitch 符号和左右手约定都应从已知姿态推导测试，并至少覆盖多个 yaw；不能让生产函数与测试共享同一个错误假设。

## 6. 幻觉与诚实性发现

模型幻觉无法由 schema 或输入控制消除。D40 中，目标羊被设为静止，平地场景也没有树；模型丢失目标后仍生成了“羊跑远了”“附近都是树”等可信但可证伪的解释。

实验呈现出稳定模式：

```text
感知足够 → 动作和描述大体一致
感知丢失 → 模型用环境叙事补全缺口
```

动作结果反馈仍然是正确方向，但反馈只有在感知和结果观察足够可靠时才能纠正幻觉。未来需要把以下内容分开：

- 工具是否执行；
- 世界产生了什么效果；
- 当前观察支持哪些事实；
- 模型是在陈述观察、作出合理推断，还是编造原因。

测试中的诚实性 oracle 可以把模型断言与服务端真值比较，但服务端真值只能作裁判，不能成为生产提示。自然度仍需要真人旁观；“是否真的动了、是否真的看见、是否捏造可核实环境细节”可以自动化大半。

## 7. 可观测性发现

当前只读调试状态保留最近一次决策的**初始**观察。每次 `look_relative` 或 `move_input` 完成后，Runtime 会把新视口发给模型，但事件日志只记录动作前后姿态，不记录该工具结果中的视口。

因此事后可以知道身体怎么动了，却不能原样回答“模型在第 7 步究竟看到了什么”。下一条聊天还会覆盖调试状态中的旧初始帧。

后续实验日志至少需要：

- 用 run/tool call 的内部标识把请求、动作和结果关联；
- 保存有界、脱敏的模型可见视口快照，或保存可复原的规范化投影；
- 明确区分初始观察、工具后观察和调试接口读取时间；
- 不把密钥、原始私人聊天、服务端裁判信息或无界世界数据写入日志。

这属于实验可观测性，不是给模型增加新能力。

## 8. 当前可以下的结论

### 有证据支持

- 短动作 Tool Loop 能在真实 Paper、真实协议客户端和真实模型之间闭环运行。
- “看看并走向一只羊”不要求模型管理世界目标 ref。
- 工具后的新观察能够支持模型继续纠正方向和距离。
- 当前主要失败集中在感知几何、运动过冲、身体未稳定和跨帧表示，而不是模型完全不会调用工具。
- 感知漏报会显著增加模型编造环境原因的概率。
- 供应商工具调用、最终输出和取消语义必须进行真实端点与跨平台测试。

### 尚不能下结论

- `D01` 已经正式选择 Tool Loop，或 Grounding/Behavior 可以整体删除；
- 多轮网络延迟天然像真人，或者当前成本可以接受；
- 低层按键足以承担长距离移动，或高级移动必然等于作弊；
- 256 个最近方块是合理视口，或紧凑四元组已经是最终 schema；
- 相对坐标应被绝对坐标替代；当前只证明相对坐标存在跨帧缺口；
- AABB、多点采样或透明材质策略已经完成生产级实现；
- 动作反馈本身足以消除幻觉；
- D40 已证明 GUI、采集、战斗、多人社交、主动性或长期记忆。

## 9. 下一轮实验顺序

以下顺序优先补证据，不预先建设完整机制：

1. **让实验可复盘。**整理并发布 D40 分支；记录每次工具后模型实际收到的有界视口；固定可复制场景存档和真值查询。
2. **做坐标系 A/B。**在同一棵树周围转头、绕行、短暂遮挡再重寻，对比仅相对、稳定局部、相对加世界坐标三种投影；记录重识别率、调用数、碰撞、token 和不实陈述。
3. **补正确可见几何。**用实体 AABB、方块暴露面和透明/非完整形状场景建立可复现 fixture，再用真人第一人称画面抽查。
4. **比较两层移动。**保留短时按键，同时试做一个只消费已观察空间、持续读碰撞与视口、可被模型检查点中断的本地移动控制器。
5. **验证视口选择。**比较近/中/远分层、同质区域压缩和时间变化优先，不再只比较 20 与 256 的总数。
6. **补时序场景。**在移动期间插入玩家消息、危险声音、受伤和普通环境声，验证保序、合并、过期、硬失效与模型自主响应的边界。
7. **建立诚实性回归。**让模型报告可由独立真值判定的事实，分别统计观察、推断和虚构；生产路径继续只使用合法玩家信息。

这些实验分别为 `D01`、`D05`、`D10`–`D27`、`D35`、`D36` 和 `D40` 提供输入。任何结果进入 accepted 基线，仍需单独的合并 PR 明确结论、被否决方案以及对代码和文档的影响。
