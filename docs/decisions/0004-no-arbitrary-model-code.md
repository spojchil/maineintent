# 0004：不执行模型生成的任意代码

> 接受日期：2026-07-12。原文以 registered skills 举例；当前实验改为 V2 structured effects，仍遵守“不执行任意代码”的核心决定。

## 背景

部分 Minecraft Agent 让模型生成 JavaScript 并直接控制 Mineflayer。这种方式原型速度快，但难以保证取消、资源预算、身体冲突、效果验证、版本兼容和长期安全。

## 决定

MineIntent 不执行模型生成的任意 JavaScript、TypeScript 或系统命令。模型只能：

- 通过当前已接受、带 schema 的模型接口提交结构化效果或调用受限能力；
- 提交由确定性运行时校验和解释的有界意图/短流程；
- 提出新的接口或流程候选，候选必须经过独立验证和接受后才能启用。

## 理由

- 所有游戏动作都能声明前置条件、身体资源、超时和取消语义。
- 执行结果可以由程序和真实世界状态验证。
- 可以防止模型绕过认知边界或调用未授权能力。
- 失败和副作用能够稳定记录与恢复。

## 后果

### 正面

- 执行行为可审查、可测试、可复现。
- 同伴模型的幻觉不会直接变成任意程序行为。
- 技能可靠性可以按版本和环境统计。

### 代价与风险

- 新能力需要显式实现或经过候选验证流程。
- 开放式建造和创新方案需要足够通用的技能与短流程表达。
- 约束设计过窄会把同伴退化为固定命令集合。

## 备选方案

- 容器中执行任意代码：降低系统风险，但不能解决游戏动作取消和语义验证。
- 只允许少量固定命令：可靠但表达能力不足。

## 后续验证

Decision Contract 和 Action Runtime 详细设计需要证明：在不执行任意代码的情况下，仍能组合出首个共同活动和之后的复杂生存行为。
