# 0002：采用事件驱动的持续同伴运行时

- 状态：accepted
- 日期：2026-07-12

## 背景

Minecraft 世界持续变化，玩家也可能在 AI 行动、模型推理或闲暇期间随时聊天、改变计划或遭遇危险。传统请求—响应聊天循环和单一长期任务循环都无法表示这种持续并发情境。

## 决定

MineIntent 使用事件驱动的 Companion Runtime。Minecraft、玩家、动作、模型、生命周期和记忆变化被规范化为有来源和因果关系的领域事件；当前对话、共同活动、意图和动作状态通过投影维护。

模型只在具有社交、活动或策略意义的事件出现时运行。实时反射、持续动作、模型决策、语言发送和后台记忆整理使用不同运行通道。

## 理由

- 能在玩家随时介入时正确打断或调整行为。
- 避免模型逐 tick 轮询。
- 使语言、行动和结果处于同一条可追踪因果链。
- 支持死亡、断线和重启恢复。

## 后果

### 正面

- 同伴可以在没有显式任务时持续存在并响应世界。
- 可独立测试事件归一化、状态投影和触发策略。
- 调试时可以回溯一次行为的输入与结果。

### 代价与风险

- 需要明确事件粒度，避免把每 tick 数据写入长期日志。
- 并发、过期决策和事件顺序需要严格语义。
- 事件投影恢复必须与服务器当前真实状态核对。

## 备选方案

- 纯聊天轮次：无法覆盖持续动作和游戏主动事件。
- 永久自我提示循环：成本高，容易产生无意义动作和人格漂移。
- 固定行为树：适合局部控制，不适合开放社交理解。

## 后续验证

首个纵向原型应覆盖聊天到达、动作执行、玩家中途改变计划、动作结果、危险反射和重启恢复的完整事件链。
