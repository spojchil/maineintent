## 1. 使用方式

每项可以处于以下状态：

- `待修正`：事实、代码行为或论证存在不一致，不需要产品取舍即可修正。
- `待决策`：存在两个以上合理方案，需要明确选择。
- `已决定`：结论、理由和后果已经记录。
- `已验证`：结论已经通过代码、测试或真人观察验证。
- `已废弃`：问题因其他决定而消失。

建议按 `D01 → D02 → D03 → D05 → D08 → D14 → D18` 的顺序先确定根边界。下游条目可等根边界确定后再审议。

## 2. 事实或逻辑矛盾

以下项目默认状态均为 `待修正`。它们不直接决定采用哪套架构，但需要在反思文档成为实施基线前得到准确表述。

### F01 — 输入控制不消除模型幻觉

草案称“模型不可能引用它从未被告知的东西”。实际只能保证模型无法从接口获得未公开信息的真实值；模型仍可编造对象、方向和参数。幻觉应由动作结果纠正。

### F02 — 并非所有具身环节失败都会丢失 speech

Decision Protocol 整体解析失败会丢弃全部 effect；Grounding 或 Behavior 拒绝后，`timing: now` 的 speech 当前已经可以独立落地。草案需要区分协议级失败和具身 effect 拒绝。

### F03 — `ActionOutcome` 同时声称“不判断成功”和“意图达成”

草案把 `completed` 定义为“意图达成”，同时又要求结果只陈述实际发生的事实。动作执行器通常只能判断动作是否终止及产生了什么效果，不能判断玩家的更高层意图是否达成。

### F04 — 世界内无效果与运行故障被混为一类

“打空”“没有挖完”可以是 `no_effect` 或 `partial`，但断线、驱动异常、非法参数、工具不可用、服务端拒绝和 Provider 超时仍需独立表达。

### F05 — `look` 并非永远完成

没有看到预期对象不是运行失败，但转头仍可能因取消、超时、断线、死亡、换维度或驱动错误而未完成。

### F06 — 当前 `withinReach` 基本冗余

现有 `lookedAtBlock` 只在约 4.5 格内射线检测，非空本身已近似表示可触达。保留的量化 `relativePosition` 仍可用于估算距离，因此用 `withinReach` 替换 `distance` 也不能阻止模型说出数字。

### F07 — `95 B → 18 B` 不全是删除 ref 的收益

18 字节示例同时删除了 `ref` 并把具名对象改成四元组。两项变化的收益需要分别计算。

### F08 — “约 1100 块”只是理论字节上限

该估算没有包含 envelope、数组分隔、tokenizer、其他 Context 内容、Provider 结果上限和模型处理大量体素的效果，不能直接作为推荐数量。

### F09 — `limit` 仍是输出与认知杠杆

`limit` 不影响当前全量扫描的 `blockAt` 调用数，但仍影响序列化、返回字节、token、模型注意力和噪声比例。“不是瓶颈”只能限定为“不是扫描耗时瓶颈”。

### F10 — 距离分层与显著性排序属于新机制

两者会改变模型获得哪些观察，是感知选择策略，不是无条件的字段替换，也不能在未验证前声明为所有路线都必须进行。

### F11 — 安全 Tool Loop 尚未完成

当前已有 Information Runtime、Tool adapter 和宿主预算，但还缺少供应商 Tool Calls 循环、`reasoning_content` 回传、`tool_call_id` 关联、动作串行化、用户中断、最终输出阶段、重试和幂等。

### F12 — 两个受众尚未形成两个明确契约

草案说明 Runtime 需要精确结果、模型需要知觉等价投影，但类型仍只有一个 `ActionOutcome`。投影位置和两个 schema 尚未定义。

### F13 — 启用 Information Tool 与退役 Cursor Store 存在冲突

如果 `information` 保留分页，Tool Loop 上线后 Cursor Store 会成为生产依赖。只有决定取消模型分页后，才能判定它可退役。

### F14 — 世界目标 ref 退出不能推出整个 Ref Store 退役

GUI 槽位、tooltip、screen 控件、分页和 Runtime 内部锁定仍可能需要 selector。需要区分模型可见的世界目标 ref 与 Information Runtime 内部 selector。

### F15 — 手持物、快捷栏和完整背包被混为同一种被动信息

当前手持物和快捷栏持续可见；完整背包通常需要打开界面或依靠记忆。`inventory_information` 不能整体按本体感觉处理。

### F16 — 主动转动视角与读取当前画面被混为一件事

`look` 主动改变视角方向，但玩家会持续看到当前画面。是否要求模型额外调用 `viewport_information` 是推理调度选择，不是视觉方向本身的要求。

### F17 — 模型网络延迟不天然等于真人感

认知迟疑可以自然；转头、瞄准、避障和危险反应等感知运动循环仍需要低延迟本地控制。需要区分认知延迟和运动控制延迟。

### F18 — Pathfinder 与 Bot 行为之间没有必然因果

机器人感也可能来自未观察地图、机械轨迹、完美转向和永不迷路。路线规划本身不是作弊，人类同样会规划路线。

### F19 — `dig`/`attack` 与“准星 + 左键”原则有张力

草案一方面把原版输入归纳为准星和左右键，另一方面按准星目标类别注册 `dig` 和 `attack`。需要把它作为接口选择，而不是已经从玩家输入自然推导出的唯一结论。

### F20 — 长动作与“玩家可能说话就太粗”判据有张力

移动 16 格或连击 10 次可能持续数秒。若它们可以被玩家消息即时中断，判据应改成“动作必须可中断”；否则这些工具自身已经过粗。

## 3. 与当前接受基线的冲突

本节不是判定反思草案错误，而是列出采用它之后必须显式废止或修订的旧结论。

### B01 — 身体 Tool Calls 与 `CompanionDecisionV2`

当前模型输出语义 effect，经 Dispatcher、Grounding 和 Behavior 执行；草案改为模型调用身体动作工具。两者不能同时作为唯一权威接口。

### B02 — Information Tool Loop 与单次被动 Context

当前系统设计、Information Runtime 文档和生产测试固定为无 Tool Loop 的单次被动观察；草案准备恢复 Tool Loop。

### B03 — 模型无 ref 与当前 viewport schema

当前 viewport 为脚下方块、准星方块、实体和可见方块发放 opaque ref；草案要求这些 ref 退出模型载荷。

### B04 — 反馈验证与当前目标授权原则

当前 Grounding 在动作前完成目标授权、作用域和证据校验；草案将边界移动到输入控制、准星选择、控制器内部验证和动作反馈。

### B05 — 退役 Behavior Synthesizer 与“运行时选择方法”

当前设计让模型表达期望结果、运行时选择执行方法；草案认为当前自由度为零，应先删除该层。

### B06 — 动作工具与玩家输入清单

具身接口清单强调移动、跳跃、左键、右键和 GUI 输入类别；草案使用 `dig`、`attack`、`use` 等动作类别。

### B07 — Pathfinder 默认取向

当前系统设计规定 Pathfinder 既不默认禁止也不默认可信；草案明显倾向于不用 Pathfinder。

### B08 — `Action Runtime` 的文档与职责

旧实现已经删除，但新动作工具仍需要 action identity、epoch、资源互斥、取消、幂等、临时目标锁和结果记录。需要判断是取消概念，还是以更小的内部控制面重建。

### B09 — “§1–§7 已达成方向”的范围

反馈闭环原则可能已经形成倾向，但身体工具类型、错误语义、视觉输入、最终模型输出和内部控制面仍未冻结。文档状态需要准确限定。

## 4. 根架构与模型协议决策

### D01 — 是否正式采用“动作—观察—再决策”的反馈闭环？

候选：完全替代 semantic goal/Grounding；与旧路径并行实验；仅用于部分动作。

### D02 — 原生 Tool Calls 负责哪些能力？

候选：只负责 Information；Information 与身体动作；包括聊天、记忆和注意力在内的所有 effect。

### D03 — Tool Loop 结束时如何输出社交和认知结果？

需要确定 speech、activity、memory 和 next attention 使用最终结构化输出、独立工具、自然语言 `content`，还是混合方式。

### D04 — 单个动作参数非法时，其他 effect 是否继续？

候选：按 effect 独立校验；把工具错误回传模型并允许修正；整轮失败。

## 5. 引用与内部控制决策

### D05 — 模型可见 ref 删除到什么范围？

分别决定世界方块、实体、消息 eventId、记忆 ID、GUI selector、分页 cursor 和供应商 `tool_call_id`。

当前已表达的方向是：模型不管理世界目标 ref；Runtime 内部标识仍保留。具体范围尚未冻结。

### D06 — Runtime 内部动作 envelope 包含什么？

候选字段包括 actionId、toolCallId、session/world/connection epoch、开始与截止时间、取消信号、幂等键、临时目标锁、resource lease 和 journal evidence。

### D07 — 控制器可使用哪些模型不可见的精确信息？

需要确定准星选择后读取精确位置、攻击自动追踪、挖掘锁定、碰撞箱和未观察周围方块避障的边界。

---

## 6. 动作接口决策

### D08 — 左键是一个工具还是两个工具？

候选：`primary_interaction()`；`dig()` 与 `attack()`；更低层的 press/hold/release。

### D09 — 右键接口的粒度是什么？

需要覆盖短按、持续使用和释放。拉弓、举盾、进食、放置和打开界面是否统一由 `use` 表达，尚未决定。

### D10 — `look` 的参数和返回是什么？

需要确定角度单位、相对 yaw/pitch、单次最大转角、glance/aim、返回 viewport 字段以及本地精确对齐范围。

### D11 — `move` 使用距离、时间还是短时控制状态？

同时决定 sprint、sneak、strafe 和自动跳跃。

### D12 — 工具最大持续时间和中断规则是什么？

需要定义玩家聊天、危险、断线和新决策到来时的 immediate/checkpoint 行为及其结果。

### D13 — 一次模型响应允许几个身体工具？

需要决定是否强制每个子轮最多一个身体动作、信息查询能否并行、多余 Tool Calls 如何处理。

## 7. 动作结果决策

### D14 — 如何区分执行终止与世界效果？

候选为双轴结构，例如：

```text
termination: completed | interrupted | timed_out | rejected | error
effect: changed | partial | none
observed: ...
```

### D15 — Runtime 精确结果如何投影为模型结果？

需要决定使用两个类型，还是一个内部 envelope 加一个明确的 model projection。

### D16 — 哪些数值可以向模型公开？

逐项决定相对位置、距离带、血量、物品数量、移动距离、挖掘进度、命中次数和动作持续时间。

---

## 8. 信息与视觉决策

### D17 — 每次模型调用自动注入哪些信息？

候选包括当前 viewport、手持物、快捷栏、HUD、最近声音、完整背包、当前 screen 和最近动作结果。

### D18 — viewport 是被动当前帧还是主动查询？

可选混合方案：每轮自动注入小型当前帧；`information` 请求更多字段；`look` 后自动返回新帧。

### D19 — visibleBlocks 使用具名对象还是紧凑四元组？

需要权衡 token、可读性、schema 演进和模型解析稳定性。

### D20 — 一次向模型提供多少方块？

20、200、1000 都需要通过真实 token 成本和模型任务表现验证。

### D21 — 方块选择策略是什么？

候选包括最近优先、距离分层、静态显著性、视线中心、玩家痕迹和时间变化优先。

### D22 — 感知层可以进行哪些确定性语义聚合？

候选包括连通区域、地形高度、材质分区和跨帧变化。拒绝额外 LLM 散文生成器，不自动等于禁止结构化聚合。

### D23 — `lookedAtBlock` 表示多远的准星射线？

候选：只报告约 4.5 格交互目标；报告更远的首个可见方块并附可触达状态；拆分近处交互目标与远处视觉目标。

### D24 — viewport 扫描采用什么性能策略？

需在真实 backend 测量后选择分壳扫描、配额早退、增量缓存、区块索引或降低调用频率。

---

## 9. 导航与身体自然度决策

### D25 — 长距离导航采用什么方案？

候选：连续短距离 move；基于已观察空间的局部地图；受限 Pathfinder；模型选择局部 waypoint；混合方案。

### D26 — 自动追踪、自动跳跃和自动瞄准允许到什么程度？

需要划定正常感知运动协调与机器人超能力的边界，并通过旁观测试验证。

### D27 — 用什么指标判断动作像真人？

至少考虑转头轨迹、停顿、路线、撞墙调整、连击节奏、玩家消息响应和错误后的自我修正。

---

## 10. 主动性、安全与记忆决策

### D28 — 主动机会由谁触发？

候选：模型自行安排；Runtime 空闲计时；Runtime 发布情境机会、模型决定说话、行动或沉默的混合方案。

### D29 — 哪些变化构成主动机会？

候选包括玩家靠近/离开、重复劳动、新建筑、天黑、危险声音、长时间无交流和共同活动进展。

### D30 — 危险反射可以执行哪些动作？

需要决定释放输入、后退、举盾、逃跑、进食及是否允许绕过模型延迟；阈值还需选择固定、配置化或组合判断。

### D31 — 具身能力与长期记忆哪个先做？

候选：动作闭环先行；记忆先行；两条最小纵向链并行。

### D32 — 第一版长期记忆必须具备哪些能力？

候选包括事件、偏好、约定、跨会话检索、整合、冲突修正、遗忘以及玩家查看、修改和删除。

---

## 11. GUI、多人和供应商决策

### D33 — GUI 是否继续使用 selector/ref？

候选：screen-bound selector；可见槽位索引；屏幕相对位置；组合方案。

### D34 — Information Tool 是否保留分页？

保留分页则 Cursor Store 仍是运行依赖；取消分页则每个接口都必须设计固定、有界的单次结果。

### D35 — 如何处理其他玩家？

需要决定可见聊天、回应范围、主要玩家记忆隔离，以及谁可以暂停或影响同伴。

### D36 — DeepSeek Tool Loop 使用什么供应商配置？

需要决定 strict/普通模式、thinking 开关、`reasoning_content` 生命周期、调用预算、`max_tokens`、空 content 重试和最终 JSON 输出。

---

## 12. 文档、迁移与验收决策

### D37 — 反思草案何时成为新权威基线？

若采纳，需要同步修订系统设计、模型接口参考、Information Runtime、ADR、no-tool 测试和 v0.3 计划。

### D38 — Grounding/Behavior 何时删除？

候选：Tool Loop 原型验证后删除；新旧路径短期并存；只删除模型协议层并保留内部控制能力。

### D39 — 如何标注文档与实现状态？

候选状态包括已实现、已设计、实验中和已废弃；另需判断是否值得建立 CI 检查。

### D40 — 新架构的最小验收实验是什么？

候选首个场景：

```text
玩家说“看看那只羊”
→ 模型获得当前视野
→ 调用相对转头
→ Runtime 返回新视野
→ 未对准则再次调整
→ 对准后自然回应
→ 全程不由模型管理世界目标 ref
```

验收需要同时观察接口正确性、动作自然度、失败恢复、响应延迟和玩家观感。
