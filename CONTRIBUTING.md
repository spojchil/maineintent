# MineIntent 贡献规范

## 基本原则

- 以[产品设计](./docs/product-design.md)为长期目标，以代码、测试和真实运行证据为当前事实。
- 优先完成可观察、可失败的小型纵向实验，不先建立尚无真实使用者的完整抽象层。
- 模型输出不是游戏事实；成功、失败和观察结论必须由实际结果支持。
- 现有设计不成立时可以直接提出替代实验，不必先把所有下游问题在纸面上回答完。
- 不提交密钥、访问令牌、私人聊天、玩家数据、世界存档、服务器地址或第三方研究仓库。

## 协作语言

- 人工创建的 Issue、Pull Request 和审查交流使用简体中文。
- 代码标识、API 名称、技术术语和 Conventional Commit 类型可以使用英文。
- Commit 说明部分优先使用中文，例如 feat: 增加相对转头工具。
- 自动化生成的内容和引用的上游错误信息可以保留原文。

## 工作方式

项目使用 GitHub Flow：

1. 从最新 main 创建短期分支。
2. 用 Issue 或 PR 说明现实场景、实验假设或要修复的问题。
3. 让每个 PR 保持一个可理解、可回滚的范围。
4. 记录实际执行过的验证，不把计划写成已经完成。
5. 通过自动检查后合并，并删除短期分支。

常用分支名：

~~~text
feat/<issue>-<name>
fix/<issue>-<name>
docs/<issue>-<name>
refactor/<issue>-<name>
experiment/<name>
~~~

较大的工作可以使用父 Issue 和 sub-issues。Milestone 应表示可验收的产品增量，不作为长期想法清单。

## Commit

使用简化的 Conventional Commits：

~~~text
feat: 增加短时移动输入
fix: 在取消后释放前进键
docs: 记录 D40 首轮观测
test: 覆盖移动中的玩家打断
refactor: 分离工具描述与执行
chore: 更新开发依赖
~~~

提交应保持可理解，不提交明显损坏的中间状态。PR 内可以有多个开发提交，合并时使用 Squash Merge。

## Pull Request

PR 应说明：

- 要观察或解决的真实问题；
- 本次实际改了什么；
- 运行了哪些检查或游戏内实验；
- 哪些结果已经观察到，哪些仍然未知；
- 是否影响接口、存储、玩家体验或兼容性。

架构结论可以来自实验，但实验实现本身不会自动成为最终架构。需要保留的长期决定进入 [docs/decisions/](./docs/decisions/README.md)；短期实验事实放在对应 PR、测试或精简实验页中。

## 验证

提交前至少运行与改动相关的检查：

~~~powershell
pnpm check
pnpm test
~~~

涉及 Minecraft 行为时，还应记录：

- Minecraft、Paper 和 Mineflayer 版本；
- 场景初始条件；
- 模型实际可见的信息；
- 玩家能观察到的动作过程；
- 动作后的世界状态；
- 延迟、模型调用次数和 token；
- 失败、取消、打断与清理路径。

不能用模型输出“成功”代替游戏状态验证。

## AI 辅助开发

允许使用 AI 编写和审查代码，但提交者仍对结果负责：

- 阅读并验证 AI 生成的实现。
- 不把生产密钥、私人聊天、玩家数据或未公开世界文件发送给外部模型。
- 不让 AI 撰写的大量文档替代实际运行。
- 大范围改动按自然职责拆分成可审查的提交和 PR。
- PR 描述只写能够由代码、测试或真实观察支持的结论。

## 文档

主线只保留当前有直接用途的文档：

- 产品目标；
- 当前事实；
- 正在进行的关键实验；
- 仍然有效的少量 ADR；
- 运行与验证指南。

过去的设计和调研保留在 Git 历史中，不在默认分支重复归档。需要恢复旧思路时，先核对当前代码和当时被放弃的原因，再决定是否重新写成当前文档。

## 安全报告

不要在公开 Issue 中提交密钥、访问令牌、服务器地址、私人日志或可利用漏洞的敏感细节。安全问题使用仓库的私密漏洞报告渠道。
