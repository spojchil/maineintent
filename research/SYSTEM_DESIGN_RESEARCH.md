# MineIntent 系统设计调研

> 状态：第一轮  
> 日期：2026-07-12  
> 目的：研究相邻项目的系统设计，为 MineIntent 的正式系统设计提供输入。本文不是最终架构决定。

## 1. 调研对象

本轮选择四个互补方向：

| 项目 | 主要方向 | 与 MineIntent 的相关性 |
|---|---|---|
| OpenClaw | 长期运行的个人 Agent、消息接入、工具、会话和记忆 | 运行时与 Agent 基础设施 |
| SillyTavern | 人格、用户设定、角色扮演、世界信息和上下文编排 | 同伴档案与关系连续性 |
| Mindcraft | LLM、Mineflayer、聊天、动作与多 Agent 协作 | Minecraft 中的语言—行动闭环 |
| Voyager | 自动课程、迭代执行、批评器和技能积累 | 长期能力成长与任务验证 |

相关但本轮未深入源码的项目还包括 GITM、MineLand、VillagerAgent 和 MineNPC-Task。它们分别涉及文本知识与记忆、有限感知下的多智能体生活、协作任务分解和记忆型 NPC 评测，可在正式设计遇到对应问题时继续研究。

第三方仓库浅克隆在 `research/repos/`，该目录不进入 MineIntent 的 Git 历史。

## 2. OpenClaw

### 2.1 它解决的问题

OpenClaw 是长期运行的个人 Agent 平台。它把不同消息渠道、会话、模型运行、工具执行、持久化、记忆和插件组织在统一运行时中。

它的基本 Agent turn 是：

```text
接收消息
→ 定位会话
→ 排队并取得会话写锁
→ 组装上下文和系统提示词
→ 调用模型
→ 执行工具
→ 流式发布模型、工具和生命周期事件
→ 写入会话记录
→ 必要时压缩上下文并重试
```

每个会话的运行被串行化，避免同一会话中多个模型调用和工具执行同时修改状态。消息可以按 steer、follow-up、collect 或 interrupt 等方式进入队列，而不是把所有新消息都当成下一轮普通聊天。

### 2.2 值得借鉴

#### 会话与运行分离

一次模型运行有独立 `runId`、生命周期、超时和结果；会话则持久保存连续历史。对 MineIntent 而言，“一次思考”和“同伴在这个世界中的长期存在”也必须分开。

#### 明确的事件流

OpenClaw 区分 assistant、tool 和 lifecycle 事件。MineIntent 也应区分：

- 玩家与游戏输入事件。
- 同伴语言输出。
- 意图和决策事件。
- 动作开始、进展、结束和失败。
- 生命周期与恢复事件。

这样聊天、游戏行动、调试界面和持久化不需要依赖一个巨大回调函数。

#### 新消息不是只有“等待”一种处理方式

正在执行动作时收到玩家消息，需要判断它是：

- 立即打断当前行动。
- 调整当前行动。
- 补充信息，当前行动继续。
- 稍后回应。
- 聚合多条消息后统一处理。

这是 MineIntent 处理自然交流的核心问题之一。

#### 上下文是可观察的产品对象

OpenClaw 能检查提示词各部分的大小、来源和是否截断。MineIntent 也需要能解释某次决策使用了哪些同伴档案、记忆、世界观察和聊天，否则很难调试人格漂移或错误记忆。

#### 稳定上下文与按需上下文分离

OpenClaw 将稳定的身份、用户和工作区信息与按需检索的详细记忆分开。只有技能摘要常驻上下文，详细技能说明按需读取。这适合 MineIntent：

- 同伴档案和当前共同活动保持紧凑且常驻。
- 详细共同经历、地点和旧事件按需检索。
- 游戏知识和技能文档只在相关时加载。

#### 记忆分层与整理

OpenClaw 区分精炼的长期记忆、详细日记和原始会话，并在上下文压缩前做一次记忆保存。它还强调记忆不等于硬权限策略，来源、有效期和行动条件需要保留。

这支持 MineIntent 将以下内容分开：

- 当前活动工作记忆。
- 共同经历流水。
- 经整理的长期关系记忆。
- 世界事实和地点。
- 临时承诺与待办。

### 2.3 不应直接照搬

- OpenClaw 的基本节奏仍以“收到一条消息后运行一次 Agent”为中心；Minecraft 同伴还需要由游戏事件和持续活动驱动。
- 普通数字工具通常是离散调用，Minecraft 动作则持续数秒到数分钟，并不断受到世界变化影响。
- 文件式自由记忆适合个人助手，但空间位置、物品、实体和正在执行的活动需要更强的结构化状态。
- OpenClaw 的高权限计算机工具安全模型不能直接等同于游戏中的社交和世界行为边界。

## 3. SillyTavern

### 3.1 它解决的问题

SillyTavern 是面向角色对话的 LLM 前端。它的核心价值不在 Agent 工具执行，而在于把角色、用户、场景、世界信息、示例、记忆和聊天历史组织成可编辑的上下文。

实际提示词可包含：

- 系统提示词。
- 角色描述、人格与场景。
- 用户 Persona。
- World Info / Lorebook。
- 示例对话。
- 作者注释和扩展提示词。
- 记忆摘要与向量检索结果。
- 最近聊天历史。

World Info 可以根据关键词或向量相似性在相关情境下激活，而不是把整个世界设定永久塞进上下文。

### 3.2 值得借鉴

#### 同伴档案是独立、可编辑和可移植的对象

MineIntent 的同伴档案也应独立于模型供应商、Minecraft 存档和运行日志。它可以包含自然语言人格、关系背景、表达倾向和示例互动，而不是被拆成许多行为参数。

#### 玩家自身也需要一个 Persona

只有 AI 人格，没有玩家档案，会迫使模型从聊天中反复猜测玩家身份和偏好。MineIntent 可以维护一个由玩家明确内容和真实互动逐渐形成的玩家侧档案，但必须允许查看和纠正。

#### 信息按情境激活

Minecraft 世界和共同经历会快速增长。类似 Lorebook 的按需激活思想适用于：

- 提到“旧基地”时检索对应地点和共同经历。
- 接近某区域时加载当地重要事实。
- 再次遇到某玩家时加载相关关系记忆。
- 开始下界活动时加载相关约定和历史失败。

#### 提示词组成必须透明

用户可理解同伴为什么表现成现在这样，并能修改档案、世界背景或错误记忆。这与产品设计中的“记忆可查看、纠正与删除”一致。

### 3.3 不应直接照搬

- SillyTavern 主要生成下一段文本，不负责保证语言与真实游戏行动一致。
- 角色扮演允许共同虚构；MineIntent 的世界事实和行动结果必须基于真实游戏状态。
- Lorebook 的关键词命中适合叙事背景，但空间、时间、所有者和事实新鲜度需要额外语义。
- 一条长聊天历史不足以表示持续并发的游戏世界。
- 角色卡中的背景不能自动成为当前世界的真实记忆。

## 4. Mindcraft

### 4.1 它解决的问题

Mindcraft 将 LLM 与 Mineflayer 连接，支持游戏聊天、查询命令、动作命令、生成 JavaScript 行动、多 Bot 对话、部分自主目标、模式反射和简单记忆。

其主要闭环是：

```text
收到玩家聊天或系统提示
→ 将状态、背包、命令文档、摘要记忆和示例加入提示词
→ 模型输出聊天或一个命令
→ 执行命令
→ 将动作结果写回历史
→ 模型继续回应或选择下一命令
```

系统还会在新消息到达时取消旧生成，并允许停止挖掘、采集、寻路和战斗。多 Bot 对话根据双方是否忙碌决定立即回应、延迟回应或忽略。

### 4.2 值得借鉴

#### 语言与行动放在同一回合

模型可以先说一句符合情境的话，再发起游戏动作；动作结果会重新进入对话。这比把聊天模型和任务 Bot 分离更接近 MineIntent。

#### 动作繁忙状态影响对话调度

Mindcraft 已意识到建造、移动、查询和战斗期间的回应策略不同。MineIntent 应进一步推广到玩家消息、危险事件、动作可中断性和说话优先级。

#### 状态按需加入提示词

背包、附近方块、实体、当前动作和相关技能文档不必全部永久注入。MineIntent 应保留这种按需观察思想，但用更明确的情境构造器控制真实性和规模。

#### 真实动作结果回到模型

示例中动作失败后，模型能够解释并选择替代行为。MineIntent 必须保留这一闭环，并让结果结构化、可验证。

### 4.3 局限和风险

- 主循环仍强烈围绕聊天命令和任务完成，陪伴与共同活动不是一等状态。
- 简单摘要字符串和地点字典无法支撑长期关系与世界记忆。
- 模型可以生成任意 JavaScript；即使放入容器，也难以进行可靠取消、预算控制和效果验证。
- 许多游戏策略固化在 modes 开关和命令中，容易让行为表现与人格决策分离。
- 对话、动作和历史集中在单个 Agent 类中，扩展并发事件与恢复会变得困难。
- 通过提示词要求模型说话和调用命令，但没有稳定的结构化“共同活动”与“短期意图”表示。

Mindcraft 适合作为快速原型和 Mineflayer 行为参考，不适合直接成为 MineIntent 的系统骨架。

## 5. Voyager

### 5.1 它解决的问题

Voyager 关注开放世界中的终身学习，而不是陪伴。它把系统拆成：

- Curriculum Agent：根据当前能力提出下一项任务。
- Action Agent：生成可执行程序。
- Critic Agent：根据执行事件判断任务是否成功并给出批评。
- Skill Manager：把成功程序保存为技能，并按语义检索复用。

一次迭代是：

```text
选择任务
→ 检索相关技能
→ 生成程序并执行
→ 获取游戏事件
→ Critic 判断成功与否
→ 失败则带批评重试
→ 成功则沉淀为可复用技能
```

### 5.2 值得借鉴

#### 执行者与验证者分离

AI 生成行动，不代表行动成功。MineIntent 应由动作运行时和世界状态验证结果；复杂活动还可以有独立评估过程。

#### 技能库是经验，不只是预装 API

长期运行后，同伴可以积累在特定世界或版本中验证过的行为经验。相关经验按语义检索，而不是把全部技能放进提示词。

#### 失败反馈进入下一轮

失败需要保留真实原因、环境变化和已产生的副作用，供下一轮调整，而不是简单返回 false。

### 5.3 不应直接照搬

- 自动课程的目标是扩大能力覆盖，不等于陪伴玩家；MineIntent 的活动来源首先是共同情境。
- 生成并执行任意 JavaScript 与 MineIntent 的长期可靠性目标冲突。
- Critic 主要依赖模型判断，而 MineIntent 应尽量使用确定性世界状态验证。
- Voyager 的世界常被测试框架重置或修改；真实陪伴世界不能依赖管理员恢复。
- 学到的技能如果未经版本、前置条件、副作用和取消语义约束，长期复用会产生风险。

## 6. 第二轮跨类型调研

第一轮偏重通用 Agent 基础设施和 Minecraft 任务 Agent。为覆盖“朋友陪伴”本身，第二轮增加了社会模拟、AI 伴侣、长期记忆、主动行为和其他 Minecraft 自主实现。

### 6.1 Generative Agents 与 AI Town

Generative Agents 研究和 AI Town 实现关注一群角色如何在持续世界中生活、相遇、交谈、形成记忆并进行反思。其重要思想不是某个具体模型，而是：

```text
观察
→ 写入情景记忆
→ 按相关性、近期性和重要性检索
→ 在积累达到阈值时形成高层反思
→ 使用记忆与反思制定计划和生成行为
```

AI Town 的源码会把一次对话从每个角色自己的视角总结为记忆，记录参与者和情感倾向，为记忆计算重要性并生成向量。检索时先扩大候选集，再综合：

- 语义相关性。
- 最近访问时间。
- 记忆重要性。

积累足够多的重要事件后，系统从多条具体记忆中形成高层反思，并保留反思依赖的原始记忆。

对 MineIntent 的价值：

- 共同经历应以 AI 自己的视角保存，而不只是聊天摘要。
- “玩家是谁、我们关系如何”可以从多个真实事件中逐渐形成，但必须保留证据并允许纠正。
- 记忆检索需要时间、重要性、关系对象、地点和当前活动等信号，不应只做向量 Top-K。
- 主动行为可以来自计划和世界情境，不必等待玩家消息。

不能直接照搬：

- 社会模拟允许模型生成世界行为；Minecraft 的世界变化必须由真实动作执行。
- 反思生成的是推断，不是事实，应明确标为推测或关系理解，不能覆盖原始事件。
- AI Town 的角色活动相对离散，Minecraft 战斗和移动需要更快的控制层。
- “重要性”由一次模型评分决定容易漂移，MineIntent 需要结合可观察事件和玩家反馈。

### 6.2 Google DeepMind Concordia

Concordia 将生成式社会 Agent 设计成可组合组件，例如：

- 指令和身份背景。
- 最近观察。
- 关联记忆。
- 当前情境判断。
- 可用选项判断。
- 根据目标选择最佳选项。
- 情绪立场和最终行动选择。

它还提供专门的 AI companion prefab，会先询问“当前对话是什么情境、最近有哪些重复或僵化模式”，再考虑可选行为和最佳行为。这说明陪伴系统不应只生成回复，还应显式维护：

- 对当前关系和互动阶段的理解。
- 最近交流是否重复、打扰或失去新鲜感。
- 此刻可以说话、沉默、提问或采取游戏行动等多种选择。

对 MineIntent 最有价值的是“认知组件可组合”，而不是采用多次串行模型调用。第一版可能需要把多个判断压缩到一次模型输出，以控制延迟和成本。

### 6.3 Letta / MemGPT

Letta 把 Agent 设计为有持久状态的实体，而不是一次性聊天请求。其核心思想包括有限的核心记忆、可检索档案记忆、上下文窗口管理，以及由 Agent 主动读写记忆。

对 MineIntent 的价值：

- 同伴身份、关系核心和当前共同活动需要稳定存在，不应每轮从完整历史重新推导。
- 模型应能提出记忆写入或修订，但写入流程必须验证来源并保留审计。
- 上下文容量是系统资源，需要明确预算和换入换出机制。

不能直接照搬：对话型 Agent 的记忆块仍不足以表示不断变化的 Minecraft 世界；空间、实体、物品和动作状态需要结构化存储。

### 6.4 memU

memU 将记忆保留为三层表示：

- File：综合后的主题记忆或技能文档，是返回给 Agent 的主要单位。
- Segment：用于向量检索的细粒度片段。
- Resource：原始来源，可在摘要不足或需要核实时回溯。

它还把对话提炼为人物资料、关系、偏好、习惯、目标、活动、观点、经历和知识等不同主题，并能把工具执行轨迹整理成经验。

对 MineIntent 的价值：

- 检索摘要时必须保留回到原始事件的路径。
- 关系记忆、世界事实和技能经验不应混在同一个向量集合中。
- 记忆整理和记忆检索可以在主同伴模型之外运行。
- 技能经验可以来自实际动作轨迹，但不应自动变成可执行代码。

### 6.5 OpenAkita 的主动行为

OpenAkita 提供主动问候、任务跟进、空闲聊天和晚安等主动引擎，并根据反馈调整频率。其价值不是这些固定场景，而是把“主动行为”视为独立调度问题，而不是在每轮提示词中写一句“请主动”。

MineIntent 可以借鉴：

- 主动机会由游戏情境和关系状态产生候选。
- 模型决定候选是否适合当前时刻以及如何表达。
- 玩家明确或隐含的负反馈影响以后主动程度。
- 主动行为需要冷却、去重和抑制机制。

但不能把陪伴变成定时消息系统。Minecraft 中的自然节点，例如完成工作、长途赶路、天黑、回到基地和发现新地点，比固定时间间隔更重要。

### 6.6 Minecraft Agent Swarm

该项目是另一种较新的 Mineflayer 自主实现，采用每个 Bot 一个事件驱动 Brain：

- 空闲定时触发战略决策。
- 敌人或受伤触发反应决策。
- 动作结束触发批评和后续决策。
- 聊天触发回复。
- 战斗使用更快的独立控制循环。

它还包含角色档案、共享公告、动作中止、技能可靠性统计、轨迹记录和卡死看门狗。

对 MineIntent 的价值：

- 进一步证明“不同事件触发不同决策路径”比单一聊天循环更适合 Minecraft。
- 实时战斗与慢速战略模型必须分层。
- 技能应记录实际成功率，长期失败的技能需要降级或停用。
- prompt—decision—outcome 轨迹应从第一版开始记录。

其局限仍然是任务与自动发展优先，多个固定职业 Bot 的设计不等于一个与真人建立关系的同伴。共享公告全部注入提示词也可能随规模增长造成上下文噪声。

### 6.7 商业和模组型 AI NPC

搜索还发现 Inworld、Convai 类商业游戏角色平台，以及多个 Minecraft “AI Friend/AI Companion”模组。这些产品通常强调语音、人格、情绪和实时对话，但公开资料较少，许多 Minecraft 模组只实现“NPC 加聊天 API”或结构化任务系统，无法验证长期自主生存和关系记忆。

它们仍有产品层参考价值：语音会显著增强在场感，角色动作与说话同步很重要；但在没有可审查架构和真实能力证据时，不应作为 MineIntent 系统设计的主要依据。

## 7. 综合结论

没有一个现有项目同时解决 MineIntent 的四个核心问题：

1. 像 SillyTavern 一样维持自然人格和关系。
2. 像 OpenClaw 一样可靠地长期运行、管理会话和工具。
3. 像 Mindcraft 一样把 Minecraft 聊天和行动连接起来。
4. 像 Voyager 一样验证行为并积累可复用能力。

MineIntent 需要组合这些思想，但不能直接把四个项目拼接在一起。

最重要的系统设计结论是：

> “同伴心智”和“Minecraft 行动运行时”必须解耦，但语言、意图、行动和结果必须通过同一条可追踪的事件闭环保持一致。

如果完全耦合，模型调用、Minecraft 状态和动作代码会混成难以恢复的 Agent 类；如果完全分离，聊天人格会与游戏 Bot 各自运行，失去可信的陪伴感。

## 8. MineIntent 的候选系统边界

本轮调研支持进一步设计以下系统边界，但名称和细节尚未最终确定。

### 8.1 Minecraft Runtime

负责真实连接、世界状态、实时控制、动作执行、取消、超时和效果验证。它不决定同伴人格或社交意图。

### 8.2 Perception and Event Layer

把协议数据转化为有来源、有时间、有可见性边界的事件和观察。区分底层控制可以使用的信息与同伴认知可以使用的信息。

### 8.3 Companion Runtime

维护同伴的持续存在、当前关注、共同活动、短期意图、对话状态和正在执行的行动。它根据事件决定是否需要模型运行、立即反射、排队或忽略。

### 8.4 Context Composer

为每次模型运行组合同伴档案、玩家档案、相关记忆、当前共同活动、最近事件、必要世界观察和可用能力摘要，并保留每一部分的来源与预算。

### 8.5 Decision Interface

模型一次决策可以同时表达：

- 是否以及怎样对玩家说话。
- 对当前情境的更新理解。
- 下一项短期意图。
- 发起、调整、等待或取消某个行动。
- 需要确认的问题。
- 值得记录的记忆候选。

这不意味着一次决策必须产生所有字段，也不意味着把自然语言完全限制成僵硬表单。

### 8.6 Memory System

分别管理同伴档案、玩家明确资料、共同经历、世界事实、地点、当前活动、承诺、推测和原始事件。记忆必须保留来源、时间、可信度和是否经过验证。

### 8.7 Skill and Action System

提供可组合、可取消、有预算并返回结构化结果的游戏能力。模型调用已注册能力或受约束流程，而不是执行任意 JavaScript。成功经验可以沉淀为更高层技能，但必须经过验证和版本管理。

### 8.8 Model Gateway

隔离模型供应商，并支持不同工作使用不同模型，例如主要同伴决策、快速是否回应判断、记忆整理和低频复杂规划。

### 8.9 Journal and Telemetry

记录原始事件、上下文组成、模型运行、语言输出、意图、动作生命周期、验证结果和记忆变更，使行为可以解释和回放。

## 9. 正式系统设计需要回答的问题

下一步不应继续横向堆积项目，而应基于调研回答以下问题：

1. MineIntent 的长期主循环由哪些事件触发？
2. 玩家聊天到达时，怎样与正在执行的游戏行动并发？
3. 当前共同活动和短期意图采用什么稳定表示？
4. 模型一次输出怎样同时支持自然语言和可靠行动？
5. 哪些游戏反射不经过模型，怎样把结果通知同伴心智？
6. 什么信息进入同伴认知，什么只供底层控制？
7. 记忆的写入、验证、整理、检索和纠错流程是什么？
8. 动作失败、死亡、断线和进程重启后怎样恢复连续性？
9. 模型、动作、记忆和事件之间怎样保留因果链？
10. 第一版原型最少需要实现哪些边界，才能验证陪伴而非单纯任务执行？

## 10. 初步建议

正式系统设计可以采用“事件驱动的持续同伴运行时”作为出发点，而不是传统的聊天请求—响应服务，也不是单一长期任务循环。

第一版应优先验证：

```text
玩家聊天与行为
→ 情境更新
→ 同伴决策
→ 语言与行动
→ 动作过程中的新事件与打断
→ 真实结果
→ 后续交流与记忆
```

只有这个闭环自然、可靠，后续增加更复杂的生存技能和长期规划才会增强“同伴”，而不是只增强“自动化 Bot”。
